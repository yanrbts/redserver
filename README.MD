## 客户端网络拦截与重定向测试配置 (128 -> 129)

本配置旨在通过 **二层劫持 (L2 Interception)** 技术，将原本发送至目标节点（130）的流量，在物理层强制重定向至分析节点（129），同时保持 IP 层目标地址不变，以满足 XDP 或透明代理的审计需求。

### 1. 环境清理与初始化
在开始拦截实验前，必须清除之前可能存在的底层规则干扰，确保流量路径的可控性。

* **清除 NAT 表规则**：防止 `DNAT` 修改目标 IP，确保 IP 层原始性。
* **重置 ARP 记录**：删除旧的动态或错误静态记录，防止二层地址冲突。

```bash
# 1. 清空所有 NAT 转换规则
sudo iptables -t nat -F

# 2. 删除针对目标 130 的现有 ARP 缓存
sudo arp -d 192.168.211.130
```

### 2. 静态 ARP 劫持配置
通过静态绑定，强制内核将发往 `130` 的 IP 包封装进 `129` 的 MAC 地址帧中。

> **注意**：此步骤是劫持的核心。由于 IP 依然是 130，129 端的网卡必须开启 **混杂模式 (Promiscuous Mode)** 才能正常接收此包。

```bash
# 将目标 IP 130 强制指向 129 节点的物理 MAC 地址
# 格式: sudo arp -s <目标_IP> <劫持者_MAC>
sudo arp -s 192.168.211.130 00:0c:29:e9:fc:ea
```

### 3. 配置校验
执行以下命令确认配置已生效：

| 检查项 | 命令 | 预期结果 |
| :--- | :--- | :--- |
| **路由走向** | `ip route get 192.168.211.130` | 应显示从 `ens33` 出口，且无 NAT 标记 |
| **二层映射** | `arp -n` | `130` 对应的 MAC 应显示为 `00:0c:29:e9:fc:ea [PERM]` |
| **实时抓包** | `sudo tcpdump -i ens33 -e -n udp port 52719` | 观察 `eth.dst` 是否为 129 的 MAC，且 `IP.dst` 仍为 130 |

### 4. 常见问题排查 (Troubleshooting)
* **包发不出去**：检查 128 本机是否开启了 `rp_filter`（反向路径过滤），若开启可能拦截此类非对称流量。
* **129 抓不到包**：检查 VMware 虚拟交换机设置。在 NAT 模式下，若 IP/MAC 不匹配，虚拟交换机可能会静默丢包。建议切换至 **LAN Segment** 模式进行二层协议开发。
* **XDP 无响应**：确保 XDP 逻辑中没有硬编码校验 `iph->daddr == 129_IP`。
